# 波形仿真与观测指南 (Waveform Observation Guide)

本文档旨在指导如何使用波形查看工具（如 GTKWave 或 Vivado 仿真器）分析 `sim/tb_i2c_mpu9250/` 下的 I2C 与 AXI-Lite 模块仿真波形。

## 1. 准备工作

首先，根据 [README.md](README.md) 或 [test_plan.md](../../docs/test_plan.md) 运行对应的测试平台以生成 `.vcd` 波形文件。例如运行 I2C Sampler 核心仿真：
```powershell
cd sim/tb_i2c_mpu9250
iverilog -g2012 -o tb_jy901_sampler.vvp -f files.f
vvp tb_jy901_sampler.vvp
```
运行结束后，当前目录下会生成对应的波形文件（如 `tb_jy901_sampler.vcd`）。使用 GTKWave 打开该文件：
```powershell
gtkwave tb_jy901_sampler.vcd
```

## 2. 核心模块级仿真 (`tb_jy901_sampler.vcd`)

该仿真主要验证 I2C 主机核心状态机和采样控制器 (Sampler) 单独运行时的行为。

### 看什么信号？
将以下信号按逻辑分层添加到波形查看大纲中：
- **系统与控制**：`clk`, `rst_n`, `start` (软件或定时器发起的脉冲触发信号)
- **I2C 物理层总线**：`scl`, `sda`, 及其三态拆解信号 `i2c_sda_i`, `i2c_sda_o`, `i2c_sda_t`
- **控制状态机与标志**：顶层 `state`，`done`, `data_valid`, `ack_error`, `timeout`, `error_code`
- **采样数据输出**：`ax_raw`, `ay_raw`, `temp_raw` 等寄存器，以及 `sample_cnt`

### 怎么看？
1. **全局定位 (Zoom Fit)**：先将波形缩小到全页，寻找 `start` 信号出现由低向高跃变（脉冲起动）的时刻。
2. **I2C 协议层时序验证（局部放大）**：
    - 时钟线与数据线：放大 `start` 后的波形，观察 `sda` 是否在 `scl` 处于高电平时被拉低——这标志着发出了合法的 **START**（起始条件）。
    - 寻址与读写控制字：继续往后看前 8 个 SCL 周期。观察总线是否打出了器件地址 `0x50` 左移一位加写标志 `0`（即 `0xA0`）。
    - 从机响应：观察第 9 个时钟周期，检查从机（由 `jy901_i2c_slave_model` 模拟）是否成功将 `sda` 拉低（反馈 **ACK**）。
    - 组合动作流：顺序检查其后的时序，是否严格符合 `写寄存器基址 (0x34)` -> 发送 **RESTART** (重复起始条件) -> `发读控制字 (0xA1)` -> 连续读 26 个字节数据，并在最后一个字节返回 NACK 然后拉起 **STOP** 条件。
3. **数据解析与指示验证**：当整个通信周期结束，状态机输出的 `done` 信号产生高电平脉冲。在这个时刻，紧盯 `data_valid` 是否为 `1`。再横向对比 `ax_raw`、`ay_raw` 的十六进制数值与从机模型发出的“假数据（如 `0x1234`）”是否完全一致。最后检查 `sample_cnt` 是否加了 1。

### 为什么这么看？
- I2C 是一种对时序和电平要求极严格的串行协议，**时序是硬件逻辑设计的核心**。必须确保起始、重启、校验机制和最后的停止条件严丝合缝，这是未来在开发板上能与真实的 JY901/MPU9250 芯片建立物理层通信的先决条件。
- 重点观测三态信号 (`sda_t`, `sda_o`) 是由于 FPGA 原生逻辑无法内部传递双向信号，必须使用开漏(Open-Drain)与上拉模拟。确认 `sda_t` 切分正确，即可保证实际硬件物理电平正常。
- 观测并核对输出数据，是为了保证 FPGA 硬件解析逻辑拼接高低字节（小端模式读取，即先收低八位再收高八位）方向和拼接位宽的绝对正确。

## 3. AXI 系统级联调仿真 (`tb_axi_i2c_jy901_top.vcd`)

加入 AXI-Lite 寄存器映射层后，本仿真用于验证 PS 侧处理器 (CPU / ARM / Python PYNQ) 通过存储器映射寄存器驱动底层 I2C 引擎的行为。

### 看什么信号？
对波形列表以 AXI 通道进行逻辑分组 (Group)：
- **AXI 写通道 (Write)**：`s00_axi_awaddr`, `s00_axi_awvalid`, `s00_axi_awready`, `s00_axi_wdata`, `s00_axi_wvalid`, `s00_axi_wready`, `s00_axi_bvalid`
- **AXI 读通道 (Read)**：`s00_axi_araddr`, `s00_axi_arvalid`, `s00_axi_arready`, `s00_axi_rdata`, `s00_axi_rvalid`, `s00_axi_rready`
- **自定义寄存器 (Soft-visible)**：配置指令流如 `CTRL` (偏移量 `0x10`), `STATUS` (`0x14`), 重大控制信息如 `WORD_COUNT`, `I2C_CLKDIV`

### 怎么看？
1. **寻找 AXI 握手机制**：通过波形检索 `valid` 与 `ready` 同样为高电平重叠的一拍，这是 AXI 成功传输一次事务的前提。
2. **检查控制配置下发**：
    - 追踪 `awaddr = 0x10` (CTRL) 的写入。核对在这一拍中，`wdata` 中对应的比特(如 bit 0 的 `oneshot_start`) 是否被正确捕捉。
    - 进一步深挖，当上述写入成功后，其是否能够顺畅地打通并激发出底层的硬件 `i2c_start` 脉冲。
3. **轮询读取与状态切换**：
    - 在发起启动指令后，去寻找 `araddr = 0x14` (STATUS) 被反复读取的动作。这就是 CPU 侧正在处于“死循环轮询等结果”的过程模拟。
    - 一直向后拖动游标，直到某次读出的 `rdata` 里的 `done` 或者 `data_valid` 标志位由 0 变 1，表示采样完毕。
4. **数据打点导出**：`done` 就绪之后，马上看后续的读通道，是否紧接着向地址 `0x18`(AX_RAW)，`0x1C`(AY_RAW) 进行读取，且数据跟之前从机塞进来的预期相同。

### 为什么这么看？
- CPU 本身不包含特定的 I2C 时序认知，全靠 AXI IP 把寄存器读写翻译成硬件微操作。我们需要证明“软件写某一内存地址 -> 此 AXI IP 解析该地址并拨动正确的底层开关”整个通路的无缝映射。
- 如果 AXI 的握手没处理好（例如 `ready` 信号由于逻辑挂死而一直为低），它将直接导致挂靠在该总线上的处理器发生宕机/系统死锁。
- 确认 CPU 能够通过 STATUS.done 来跳出轮询死循环，意味着软硬件时钟域、通信状态反馈达到了健壮的预期，不仅没卡主 I2C 物理传输，也不妨碍软件层面的任务流转。

## 4. 异常及超时路径仿真 (`tb_i2c_master_timeout.vcd` & NACK 分析)

稳定可靠的硬件不只会在理想状态下工作，还必须在真实物理报错（如跳线松动或器件未就绪）时保障系统依然能全身而退。

### 看什么信号？
重点关注报错状态字和容错复位：
- **错误指示**：`timeout`, `ack_error`, AXI 层的 `ERROR_CODE`
- **测试前向干预**：`DEV_ADDR` 和人为故障源输入信号

### 怎么看？
1. **伪造无响应环境以观测超时**：在对应的 timeout 仿真中，测试平台可能缩短了配置的上限等待时间或者强制断开了线路上的响应信号。我们需要盯住 `timeout` 标志，看它在无响应达到固定拍数后是否按预期独立拉高。
2. **追踪错误捕捉与记录**：
   - 当遇到地址发送期间从机拒绝握手 (NACK，例如发送错误的地址 `0x51`) 时，`ack_error` 必须及早反馈。
   - 随之，去查看 `ERROR_CODE` 寄存器里是否立刻呈现出对应的代码（比如器件 NACK 被存为 `0x01`，总线超时存为 `0x10` 等）。
3. **总线释放与复位兜底**：发生异常后的十几个周期是最关键的。观察这之后的控制状态机，它**绝对不能死锁**，它必须能顺滑切换回 `IDLE` 等待状态并结束/回拉 `scl` 与 `sda` 到高电平。

### 为什么这么看？
- 对于真正的底层硬件设备而言，线接触不良是最常见的场景之一。如果不处理超时和 NACK 容错，一旦线路上丢失个脉冲卡住等待 ACK 的步骤，不仅这块芯片废了，连带着会让 PYNQ 卡死。
- 观测 `ERROR_CODE` 是在为顶层的 Python 开发人员提供除错支持的证据（确保 Python 可以读回来一个特定的 10 进制数来精确反推是高亮位报错还是设备寻址就失败了）。
- 确认状态机会自己归位 `IDLE`，意味着 IP 具备“自恢复能力”，下次 CPU 再重新发送一次指令，设备依然能够像没崩溃过一样再次尝试。
